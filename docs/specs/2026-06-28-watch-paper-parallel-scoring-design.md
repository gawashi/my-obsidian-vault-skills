# 設計: watch-paper の採点を per-paper 並列サブエージェント＋正規化パスに変更

- **日付**: 2026-06-28
- **ステータス**: 承認待ち（実装計画フェーズ手前）
- **対象**: `skills/watch-paper/` の **採点ステップ**を、現状の「単一エージェントが全候補を一括採点」から「**1論文=1サブエージェントの並列採点 ＋ メインによるテーマ内正規化パス**」に変更する。
- **元 spec**: `docs/specs/2026-06-28-paper-watch-skill-design.md`。本書は元 spec の **§4.4 ステップ3（採点）と §4.5（ルーブリックの適用方法）を上書きする差分**。元 spec のその他（取得・台帳・ダイジェスト書式・責務境界）は不変。

---

## 1. 背景・動機

現状、関連度スコアリングは **LLM（スキル本体＝メインエージェント）が `candidates.json` の全候補を1コンテキストで一括採点**している（元 spec §4.4 step3, §4.5）。

問題意識（ユーザー報告）:

- **候補数が多い**と1コンテキストに多数の論文が並び、1件あたりの**集中度（採点品質）が落ちる**。
- 多数を一度に捌くことで採点の**ブレ**が気になる。

要望: **並列サブエージェントで分散し、各サブエージェントを1論文に集中させたい。**

### トレードオフの明示（設計上の核心）

per-paper の並列化は「1論文への集中」を上げる一方、各サブエージェントが**他の候補を見ずに独立採点する**ため、0〜5スケールの基準が**論文間でブレやすくなる**（=横並びの一貫性は下がる方向）。一括採点は逆に、全候補を見比べるので相対的一貫性が出やすい。

本設計はこの両立のため **「per-paper 並列採点（集中）＋ メインによるテーマ内正規化パス（一貫性）」** の2段構えを採る。

---

## 2. 確定した決定事項

| 論点 | 決定 |
|---|---|
| 採点の分散 | **1論文=1サブエージェント**の並列ディスパッチ（per-paper） |
| 一貫性の担保 | サブの暫定スコアを、メインが**テーマ内で正規化（較正）**する2段構え |
| 採点の深さ | **アブストのみ**（title+abstract）。PDF本文・メタ文脈の追加取得はしない（元 spec のスコープ維持） |
| スケール制御 | 1論文1サブは保ちつつ、**同時実行数に上限 K**（既定 6）。波状にキューを消化 |
| 実装方式 | **方式 A: SKILL.md の手順としての並列ディスパッチ**（メインエージェントが Agent ツールで扇形展開）。Workflow 化（方式 B）は週1手動運用には過剰として不採用 |
| 変更範囲 | **SKILL.md のステップ2のみ**。`fetch_arxiv.py`・`tests/`・`scores.json` 契約・`--commit`・ダイジェスト書式・台帳CSVは**不変** |
| 正規化の範囲 | **テーマ内のみ**（テーマごとに anchors が異なり 0〜5 はテーマ相対のため、テーマ間では正規化しない） |
| サブ失敗時 | メインが当該1件を同じルーブリックで**インライン採点にフォールバック**（取りこぼし防止） |

## 3. スコープ

### この変更でやること

1. SKILL.md ステップ2を「単一エージェント一括採点」から「per-paper 並列採点＋テーマ内正規化」に書き換える。
2. サブエージェントの I/O 契約（入力プロンプト＋構造化出力）を SKILL.md に定義する。
3. 同時実行上限 `K` を導入する（`config.json` の `defaults.scoring_concurrency`、既定6）。
4. 正規化パス・フォールバック・報告項目（ディスパッチ数/調整数/フォールバック数）を SKILL.md に追記する。
5. README に並列採点の注記を加える。

### この変更でやらないこと

- `fetch_arxiv.py` のロジック変更（取得・コミット・CSV 書き込み）。
- `tests/` の変更（採点は LLM 側でスクリプトのテスト対象外）。
- `scores.json` のスキーマ／`--commit` インターフェース変更。
- ダイジェスト書式・台帳 CSV 列の変更。
- 採点の深さの拡張（PDF 本文取得など）。
- Workflow 化・スケジューラ連携・通知。

## 4. 詳細設計

### 4.1 作業単位

採点単位は **(テーマ, 候補) のペア**。`candidates.json` の各候補は1テーマに属する（同じ論文が複数テーマにヒットすれば候補エントリも各テーマに1つずつ存在する）。各サブエージェントは **1候補を、その候補が属するテーマの `anchors` と `threshold` に照らして**採点する。

→ 「同じ論文が複数テーマのセクションに重複掲載されうる」という元 spec の仕様（§5）はそのまま維持される（テーマごとに独立採点）。

### 4.2 サブエージェントの I/O 契約

**方式**: メインエージェントが Agent（サブエージェント）ツールで汎用サブエージェントを起動する。採点に必要なデータは**すべてプロンプトに内包**するため、サブエージェントは**ファイル/ネットワーク等のツールを使わない**（安価・隔離）。モデルは原則セッションのモデルを継承する（明示の上書きは任意の調整余地）。

**入力（プロンプトに内包する内容）**:

- テーマ: `name` / `anchors`（同種仕事の代表例）/ `threshold`
- 関連度ルーブリック（0〜5）— 元 spec §4.5 をそのまま:
  - **5**: ど真ん中。アンカーと同種のシステム/手法/問題設定。
  - **4**: 強く関連。隣接サブ問題・主要構成要素・明確な発展。
  - **3**: 関連あり。応用・部分的に主題を扱う（採用閾値の既定）。
  - **2**: 周辺。キーワードは当たるが主題が異なる。
  - **0–1**: 無関係。キーワードの誤ヒット。
- 候補1件: `arxiv_id` / `title` / `abstract` / `authors` / `published` / `primary_category` / `link`
- ガードレール: 「要約・なぜ気になるかは**アブストの範囲で書き、性能数値や主張を捏造しない**」

**出力（サブエージェントはこの JSON オブジェクトだけを返す）**:

```json
{
  "arxiv_id": "2406.xxxxx",
  "score": 4,
  "summary_ja": "（一言要約・日本語1〜2文）",
  "why_ja": "（なぜ気になるか・1行）",
  "rationale": "（このスコアにした理由・どのアンカー/バンドに当たるか・1行）",
  "confidence": "high"
}
```

- `score`: 0〜5 の整数。
- `summary_ja` / `why_ja`: ダイジェストに掲載する文面。
- `rationale` / `confidence`(`low`|`med`|`high`): **正規化パス用**のメタ情報。ダイジェストには出さない。
- メインは `arxiv_id` で結果を候補に突き合わせる。

### 4.3 オーケストレーション（SKILL.md ステップ2前半）

1. `candidates.json` を読む。`new=0` のテーマ・`error` を持つテーマはディスパッチ対象外（後述 4.6）。
2. enabled かつ `new>0` の各テーマについて、候補リストを作業キューにする。
3. **同時 K 件まで**でサブエージェントを並列ディスパッチし、波状にキューを消化する。
   - `K` は `config.json` の `defaults.scoring_concurrency` を読む。無ければ既定 **6**。
   - `scoring_concurrency` は `config.json` への**任意キーの追記**。`fetch_arxiv.py` は `defaults` の特定キーしか参照せず未知キーを無視するため、**スクリプトのコード変更は不要**。メインエージェントが `config.json` を直接読んで参照する。
4. 返ってきた結果を `arxiv_id` で突き合わせ、テーマごとに暫定スコア表へ集約する。

### 4.4 正規化パス（SKILL.md ステップ2後半・メインのみ）

サブの暫定スコアを、メインエージェントが**テーマ内で較正**する:

1. **テーマごとに**、全候補の暫定スコアをスコア順に並べる（テーマ間は混ぜない／正規化しない — anchors が異なり 0〜5 はテーマ相対のため）。
2. **境界事例**を重点的に見直す:
   - `threshold` 周辺の境目（既定 `threshold=3` なら **2/3 と 3/4** の境界）。
   - `confidence: low` の候補。
3. 対象候補について `candidates.json` のアブストを読み直し、各サブの `rationale` を材料に、**横並びの一貫性が出るようスコアを調整**する。
4. これは**再採点ではなく較正**。`summary_ja` / `why_ja` は原則サブの出力をそのまま採用し、明らかな誤りのみ軽修正する。
5. 調整した件数を記録し、報告（4.7）に出す。

### 4.5 下流（変更なし）

- 正規化後の最終スコアを **`scores.json`**（`{theme-id: {arxiv_id: score}}`）に書き出す — 契約は不変。
- ダイジェスト `digests/YYYY-MM-DD.md` を書く — 書式は不変（`summary_ja`/`why_ja` の出所がサブ＋メイン較正に変わるだけ）。
- `uv run ... fetch_arxiv.py --commit "<data_dir>/scores.json"` で台帳 CSV に追記 — フロー・列は不変。

### 4.6 エラー処理・エッジケース

- **サブエージェント失敗 / 出力が壊れている / スキップされた** → メインが**当該1件を同じルーブリックでインライン採点**してフォールバックする。論文を取りこぼして台帳から漏らさない（漏れると次回再評価でトークンを無駄にする）。フォールバック件数を報告する。
- **`new=0` のテーマ** → サブをディスパッチせず「該当なし」として扱う（元 spec 通り）。
- **取得失敗テーマ（候補に `error`）** → サブをディスパッチせず、`error` を報告に明記（元 spec 通り、全体は止めない）。
- **同じ論文が複数テーマに出る** → (テーマ, 候補) ごとに別サブを立て、各テーマの anchors で独立採点（重複掲載は仕様）。
- **候補が多い日** → K 件ずつの波で処理。ディスパッチ総数を報告する。

### 4.7 報告（SKILL.md ステップ6に追記）

従来の「テーマ別 評価件数 / 抽出件数」「上位ピック」「ダイジェストのパス」「スキップ/エラー」に加えて:

- ディスパッチした**サブエージェント数**。
- 正規化パスで**スコアを調整した件数**。
- **フォールバック採点した件数**（サブ失敗時）。

| ファイル | 変更 |

## 5. 受け入れ基準（Acceptance Criteria）

1. 採点ステップで、enabled かつ `new>0` の各テーマの各候補に対し **1論文=1サブエージェント**が起動され、**同時実行は `scoring_concurrency`（既定6）以下**に保たれる。
2. 各サブエージェントは **title+abstract のみ**を根拠に、当該テーマの `anchors`/ルーブリックに照らして 0〜5 を付け、`{arxiv_id, score, summary_ja, why_ja, rationale, confidence}` を返す。
3. メインエージェントが**テーマ内で正規化パス**を実行し、threshold 周辺の境界事例と低 confidence を見直してスコアを較正する（テーマ間では正規化しない）。
4. 正規化後の最終スコアが既存契約どおり `scores.json`（`{theme-id: {arxiv_id: score}}`）に書かれ、`fetch_arxiv.py --commit` 経由で台帳 CSV に追記される（**スクリプト・CSV 列・コミットフローは不変**）。
5. ダイジェスト `digests/YYYY-MM-DD.md` は従来書式どおり生成され、`new=0` テーマは「該当なし」、取得失敗テーマは error が明記される。
6. サブエージェントが失敗/壊れた出力を返した候補は、メインのインライン採点にフォールバックされ、台帳から漏れない。
7. 報告に「ディスパッチ数 / 正規化調整件数 / フォールバック件数」が含まれる。
8. `fetch_arxiv.py` と `tests/` は変更されない。`config.json` の変更は `scoring_concurrency` の追記のみで、既存の取得・コミット挙動に影響しない。

## 6. リスク・留意点

- **コスト増**: 1論文1サブ＋正規化で、一括採点よりトークン/起動コストは増える。`scoring_concurrency` と、テーマの `threshold`/`keywords` 調整で候補数自体を抑えることが緩和策。
- **オーケストレーションの非決定性**: 方式 A はエージェント駆動のため、波の分割やフォールバック発生は実行ごとに揺れうる。最終スコアの契約（`scores.json`）と台帳は決定的に保たれるので下流は安定。
- **正規化の効き具合**: 正規化はメインの判断に依存する。境界事例＋低 confidence に絞ることで、全件再採点せずに一貫性を底上げする方針。効果が薄ければ、サブに「アンカーとの距離」を数値で返させる等の拡張余地がある（本 v では不採用）。
