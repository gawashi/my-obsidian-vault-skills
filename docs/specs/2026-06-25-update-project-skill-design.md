# 設計: プロジェクト運用・更新スキル `update-project`

- **日付**: 2026-06-25
- **ステータス**: 承認待ち（実装計画フェーズ手前）
- **対象**: `new-project` スキルで初期化した `01-projects/` 配下プロジェクトを、進捗・要件変更・新規サブタスク等に応じて適切に更新する Claude Code スキル
- **関連**: `new-project` 設計（`.claude/docs/specs/2026-06-25-new-project-skill-design.md`）。本スキルはその §4.3「運用フロー」および §7「将来拡張: `/project-log` 的スキル」の実行版。

---

## 1. 背景・目的

`new-project` は標準構成（`CLAUDE.md` / `LOG.md` / `INDEX.md` ＋ 横断台帳 `projects.base`）でプロジェクトを**作る**。しかしプロジェクトは育つ — 進捗が出る、要件・目的が変わる、重いサブタスクが増える、外部リソースが増える、節目を迎える。これらが起きたとき「どのファイルに何を書くか」を毎回判断・記憶するのは摩擦になり、放置すると `LOG.md` が止まり `updated` が陳腐化し、台帳が現実とズレる。

**目的**: 「起きたこと」を伝えるだけで、適切なファイルへ追記・編集し、必要なら新規ノート/サブフォルダを作成し、**最後に必ず LOG/INDEX を整合させる**スキルを用意する。`new-project`（作る）の対として `update-project`（育てる）を独立させ、役割を明確にする。

### 核心方針
- 一貫性は「役割ごとに分離した少数の固定ファイル」に宿る（`new-project` と同じ哲学）。本スキルはその固定ファイル群と現実を**同期し続ける**役割。
- 1つの出来事が複数ファイルに波及してよい（例: 要件変更 → `CLAUDE.md` 概要 ＋ `LOG.md` 記録）。
- 新規ノート/フォルダを作ったら**作りっぱなしにせず** INDEX 登録＋ LOG 記録まで行い、構造の迷子を防ぐ。
- ファイル変更は**計画を提示してから適用**する（不要な往復を減らしつつ、概要書き換え・status 変更・新規作成の事故を防ぐ）。

## 2. 確定した決定事項

| 論点 | 決定 |
|---|---|
| スキル形態 | 独立した姉妹スキル `update-project`（単一 `SKILL.md`）。`new-project` とは別 |
| 対象の特定 | ハイブリッド: ①明示された名前/パス優先 → ②文脈から推測し1回確認 → ③曖昧なら候補列挙して選択 |
| 対象の前提 | `CLAUDE.md` を持つ新スキーマのプロジェクトのみ。旧 `META.md` は対象外（移行しない） |
| 扱う範囲 | フル管理: §4.2 のルーター5パターン ＋ 新規ノート/サブフォルダ作成 |
| 確認レベル | 計画（ファイルごとの変更・新規作成を箇条書き）を提示し、承認後に一括適用 |
| 新規ノート | `YYYY-MM-DD-{トピック}.md`、frontmatter は `created`/`updated` のみ（`project` タグは付けない） |
| サブフォルダ | サブタスクが複数ノート/添付を抱えるのが明らかなときだけ作成（計画に明示して確認） |
| `updated` 規則 | 編集した固定ファイルすべての frontmatter `updated` を当日日付へ。日付はハードコードしない |
| スコープ外 | 初期 scaffold（`new-project` の責務）/ 旧スキーマ移行 / `projects.base` 改変 / 完了プロジェクトのアーカイブ移動 |
| スキル置き場 | `.claude/skills/update-project/SKILL.md` |

## 3. スコープ

### v1 でやること
1. 対象プロジェクトの特定（ハイブリッド、§4.1）。
2. 対象の固定ファイルを読んで現状把握。
3. 「起きたこと」をルーティング表（§4.2）へ写像し、追記/編集を行う。
4. 重いサブタスク等で新規ノート（必要ならサブフォルダ）を作成し、INDEX 登録＋ LOG 記録で整合（§4.3）。
5. 計画提示 → 承認 → 一括適用 → 触れたファイルの `updated` を当日へ → 報告（§4.4）。
6. エッジケースの安全処理（§5）。

### v1 でやらないこと
- 新規プロジェクトの初期 scaffold（`new-project` が担う）。
- 旧 `META.md` プロジェクトの新スキーマ移行。
- `projects.base` のスキーマ変更・ビュー拡充。
- 完了プロジェクトのアーカイブ用フォルダ移動（`status: done` は付けるが移動はしない）。
- `LOG.md` / `INDEX.md` 肥大化時の分割運用。

## 4. 詳細設計

### 4.1 対象プロジェクトの特定（ハイブリッド）
1. **明示**: 呼び出し文にプロジェクト名/パスがあればそれを採用。
2. **推測＋確認**: 明示が無ければ、会話の文脈・直近で触れたファイルから対象を推測し、**1回だけ確認**する（例:「対象=`work/FHV7-AI向け良品学習アルゴ商品化支援` でいい？」）。
3. **列挙＋選択**: 曖昧/不明なら、`01-projects/{personal,work}` 配下で **`CLAUDE.md` を持つフォルダ**（＝新スキーマ）を候補として列挙し（`status: active` を上位に）、選んでもらう。
4. **前提検証**: 対象フォルダに `CLAUDE.md` があること（新スキーマ）。固定ファイルが欠けている場合は §5 のとおり処理する。

### 4.2 ルーティング表（中核）— 「起きたこと → 更新先」
`new-project` §4.3 の実行版。1つの出来事が複数行に該当してよい。

| 起きたこと | 更新先 | 操作 |
|---|---|---|
| 作業した（進捗・次アクション） | **LOG.md** | `## 現状 / 次の一手` を最新化し、`## ログ` 先頭に `- {当日}: 要点` を追記（新しいものが上） |
| 重要なノートができた/作った | **INDEX.md** | 索引に `[[ノート名]]` を追加（雑多なノートは索引しなくてよい） |
| 外部リソース（リンク/パス/リポジトリ）が増えた | **CLAUDE.md** | `## 関連リンク・リソース` に追記 |
| 目的・ゴール・**要件**の認識が変わった | **CLAUDE.md `## 概要`** を書き換え ＋ **LOG.md** に変更を1行記録 | 重要変更は両方に残す |
| 節目（着手/中断/完了/中止） | **CLAUDE.md** frontmatter `status`（`active`/`paused`/`done`/`cancelled`）＋ **LOG.md** に1行 | |
| 重いサブタスク・新しい作業領域が生まれた | **新規ノート（必要ならサブフォルダ）作成 → INDEX 登録 → LOG 記録**（§4.3） | |

**共通ルール**: 編集した各固定ファイルの frontmatter `updated` を当日日付へ更新する。日付はハードコードせず、実行時の現在日付（`currentDate` / `date +%F`）を使う。

### 4.3 新規ノート/サブフォルダの規約
- **判断基準**: 追記で済む軽い更新は §4.2 で処理。**比較的作業量の多いサブタスク・独立した作業領域**が生まれたときに新規ノートを作る。
- **新規ノートのファイル名**: `YYYY-MM-DD-{トピック}.md`（vault 既存ノート例 `2026-06-01-評価画面再設計-design.md` の流儀に合わせる）。既定はプロジェクト直下に置く。
- **frontmatter**: `created: {当日}` / `updated: {当日}` のみ。**`project` タグは付けない**（`project` タグを持つのは `CLAUDE.md` だけ → 新規ノートは `projects.base` 台帳に出ない。`new-project` §4.4 と整合）。
- **本文**: `# {タイトル}` ＋ 目的を示す1行程度の薄い雛形を seed する。ユーザーが内容を渡した場合はそれを反映。中身を勝手に膨らませない。
- **サブフォルダ**: そのサブタスクが**複数ノート/添付を抱えるのが明らか**なときだけ作成する（判断は計画に明示して確認）。名前に Windows 禁止文字 `\ / : * ? " < > |` を含めない。新規ノートはその中に置く。
- **作成後の整合（必須）**: `INDEX.md` に `[[リンク]]` を追加し、`LOG.md` に `- {当日}: {サブタスク}（{ノートへの参照}）` を1行記録。`INDEX.md` / `LOG.md` の `updated` を当日へ。
- **wikilink 注意**: `CLAUDE.md` / `LOG.md` / `INDEX.md` は vault 全体で同名のため basename wikilink が曖昧。新規ノートは固有名なので INDEX からの `[[ノート名]]` リンクは問題ない（`new-project` §5 の注意を踏襲）。

### 4.4 挙動フロー
1. 対象プロジェクトを特定する（§4.1）。
2. 対象の `CLAUDE.md` / `LOG.md` / `INDEX.md` を**読んで現状を把握**する（`status`、現状/次の一手、既存索引、既存の外部リンク）。
3. ユーザーの「起きたこと」記述（＋直近の会話）を解釈し、§4.2 のルーティングへ写像する。各項目について**追記で済むか新規作成が要るか**を判断する（§4.3 の基準）。
4. **計画を提示**する: ファイルごとに「何をどう変えるか」「新規作成するファイル/フォルダ」を箇条書きし、承認を取る。
5. 承認後に一括適用する。触れた固定ファイルの `updated` を当日へ更新。新規ノートは必ず INDEX 登録＋ LOG 記録まで完了させる。
6. 変更・作成したパスを列挙して報告し、Obsidian で開く案内を添える。

### 4.5 スキルファイル構成
```
.claude/skills/update-project/
└─ SKILL.md        ← frontmatter(name/description) + ルーティング表 + 手順
```
- `name`: `update-project`
- `description`: 「`01-projects/` の既存プロジェクトで、進捗・要件/目的変更・新しいサブタスクやノート・外部リソース追加・節目があり、ファイル更新が必要なとき。`/update-project` でも起動」を含むトリガー文。
- 既存スキル（`new-project` は単一 `SKILL.md`）の流儀に合わせる。

## 5. エッジケース・注意点
- **新スキーマでない（`CLAUDE.md` が無い）**: 勝手に scaffold せず報告する。`/new-project` での初期化を促すか、欠けている固定ファイルだけ作るかをユーザーに尋ねる。
- **対象が曖昧/複数候補**: 候補を列挙して選択させる（§4.1-3）。
- **新規ノート名の衝突**: 同名 `YYYY-MM-DD-{トピック}.md` が既存なら、連番/補足語を付けるか上書き回避をユーザーに確認する。
- **禁止文字**: ノート/サブフォルダ名に Windows 禁止文字があれば指摘し調整する。
- **出来事が写像できない/実質変更が無い**: 推測で書かず、「何を・どこに記録したいか」を確認する。
- **`updated` の陳腐化**: 「編集と同時に当日日付へ」で担保。最近の活動は `LOG.md` の `updated` が最も正確。
- **日付**: ハードコードせず実行時の現在日付を用いる。

## 6. 受け入れ基準（Acceptance Criteria）
1. `/update-project`（または「このプロジェクトに進捗を記録して」等の意図）で起動し、対象プロジェクトを §4.1 のハイブリッドで特定できる（明示があれば即採用、無ければ推測確認、曖昧なら列挙選択）。
2. 「進捗」を伝えると、計画提示のうえ `LOG.md` の `現状/次の一手` が更新され `ログ` 先頭に `- {当日}: …` が追記され、`LOG.md` の `updated` が当日になる。
3. 「要件/目的が変わった」を伝えると、`CLAUDE.md` の `概要` が書き換わり、かつ `LOG.md` に変更が1行記録され、両ファイルの `updated` が当日になる。
4. 「外部リソース追加」は `CLAUDE.md` の `関連リンク・リソース` に、「節目」は `CLAUDE.md` frontmatter `status` に反映される。
5. 「重いサブタスク」を伝えると、計画提示のうえ `YYYY-MM-DD-{トピック}.md`（frontmatter `created`/`updated` のみ、`project` タグ無し）が作成され、`INDEX.md` に `[[リンク]]` が追加され、`LOG.md` に1行記録される。必要時はサブフォルダも作られ、その中にノートが置かれる。
6. すべての変更は適用前に「ファイルごとの計画」として提示され、承認後に一括適用される。
7. 対象に `CLAUDE.md` が無い場合、勝手に scaffold せず報告し、初期化方法を案内する。
8. 旧 `META.md` プロジェクトと `projects.base` のスキーマは変更されない。

---

## 7. 将来の拡張（参考）
- 完了プロジェクトの `done` → アーカイブフォルダ移動スキル。
- 旧 `META.md` → 新スキーマ移行スキル。
- `LOG.md` を対象にした「最近の活動」横断ビュー（`projects.base` 側）。
- `LOG.md` / `INDEX.md` 肥大化時の分割運用ガイド。
- 新規ノートのタイプ別雛形（design / investigation / meeting など）の選択。
