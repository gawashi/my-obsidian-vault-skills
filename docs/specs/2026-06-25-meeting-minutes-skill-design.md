# 設計: ミーティング議事録生成＆プロジェクト紐づけスキル `meeting-minutes`

- **日付**: 2026-06-25
- **ステータス**: 承認待ち（実装計画フェーズ手前）
- **対象**: Obsidian vault の `05-meeting/` に作成したミーティングノート（トランスクリプト貼り付け済み）から、(1) 議事録を生成し、(2) 内容から該当プロジェクトを推定して `01-projects/.../LOG.md` と `INDEX.md` を更新する Claude Code スキル

---

## 1. 背景・目的

`05-meeting/` には**プロジェクトフリー**（特定プロジェクトに属さない）でミーティングのトランスクリプト＆議事録を保存している。運用は手動:

1. mtg 終了後、ユーザーが `_templates/meeting.md`（Templater）から `05-meeting/{YYYY-MM-DD}-{タイトル}.md` を作る。
2. ノート末尾の `# Teamsトランスクリプト` 以下に Teams のトランスクリプトを貼る。
3. その内容を AI に読ませて議事録（冒頭の構造化セクション＋本文）を生成する。

**自動化したいこと**: 「トランスクリプトを貼ったファイル」を渡すと、AI が議事録を生成し、**さらに該当プロジェクトの `LOG.md` / `INDEX.md` を自動更新する**。ミーティングはプロジェクトフリーなので、**内容からプロジェクトを推定し、ユーザー確認を経て紐づける**ことがこのスキルの肝になる。

### 核心方針
- ミーティングノートはこれまで通り `05-meeting/` にフラットに置く（プロジェクトフリーを維持）。
- プロジェクトとの結合は、(a) プロジェクト側 `LOG.md`/`INDEX.md` からの wikilink と、(b) ミーティングノート frontmatter の `project` リンク、の**双方向**で表現する。
- 誤ったプロジェクトの LOG を汚さないよう、**更新前に必ずユーザー確認**する（自動推定はするが自動確定はしない）。
- 既存プロジェクトの構成のばらつき（単一 `.md`、`CLAUDE.md` 欠落等）は**考慮しない**。`new-project` スキルが定める標準構成を前提とする。

## 2. 確定した決定事項

| 論点 | 決定 |
|---|---|
| 対象ファイルの特定 | 起動時に**ファイル名（または日付/タイトル）を明示的に渡す**。渡されなければ `05-meeting/` の最新更新ファイルを候補として提示し確認する（補助動線） |
| ファイル内に生成する内容 | **構造化セクション**（`会議情報` の参加者 / `議題` / `決定事項` / `アクションアイテム` / `次回の予定`）**＋ `# 議事録` ナラティブ本文** |
| トランスクリプトの扱い | **削除せず保持**。トランスクリプトが空なら中止して貼り付けを促す |
| 既存議事録がある場合 | 構造化セクションや `# 議事録` が既に埋まっていれば**上書き前に確認** |
| プロジェクト特定方法 | 内容から**自動推定 → AskUserQuestion で確認**（複数選択可・「該当なし」も選べる） |
| 推定に使う材料 | フォルダ型プロジェクトの `CLAUDE.md` の `概要` と `code`、議事録の議題・決定事項・キーワード |
| 複数プロジェクト該当 | **それぞれの `LOG.md`/`INDEX.md` を更新**（確認時に複数選択） |
| 該当プロジェクトなし | **議事録生成のみ**で終了。`LOG/INDEX` 更新はスキップしてその旨を報告 |
| `LOG.md` 更新 | `## ログ` 先頭に `- {会議日}: {1行要約} → [[mtgノート名]]`（新しいもの上）。必要に応じ `## 現状 / 次の一手` を今回の決定・アクションで更新 |
| `INDEX.md` 更新 | `## ミーティング` 見出し（無ければ作成）に `- [[mtgノート名]]` を追加 |
| 双方向リンク | mtg ノートの frontmatter に `project` を追加（**パス修飾 wikilink** で各プロジェクトの `CLAUDE.md` を指す。表示名＝プロジェクト名）。複数該当ならリストで複数持つ |
| `updated` の更新 | 編集した `LOG.md`/`INDEX.md` の frontmatter `updated` を**処理日（当日）**に更新（`new-project` の運用ルール準拠） |
| 既存構成のばらつき | **考慮しない**。標準構成（`new-project` 書式）前提。`LOG.md`/`INDEX.md` が無ければ `new-project` 書式で作成 |
| スキル置き場 | `.claude/skills/meeting-minutes/` |

## 3. スコープ

### v1 でやること
1. 対象ミーティングファイルの特定（明示渡し／最新候補の確認）とトランスクリプト存在チェック。
2. トランスクリプトから議事録を生成（構造化セクション＋`# 議事録` 本文）。トランスクリプト本文は保持。
3. ミーティング内容からフォルダ型プロジェクトを推定し、AskUserQuestion で確認（0/1/複数）。
4. 確認された各プロジェクトの `LOG.md`（ログ追記＋必要なら現状更新）と `INDEX.md`（ミーティング索引追記）を更新。不在なら `new-project` 書式で作成。
5. ミーティングノート frontmatter に `project` リンクを追加（双方向リンク）。
6. 処理結果（生成した議事録の要点、更新したプロジェクトとファイルパス、スキップ項目）を報告。

### v1 でやらないこと
- アクションアイテムを `04-daily` やタスク管理へ転記・タスク化する連携。
- トランスクリプトの話者名から参加者マスタを引いて正式名称へ正規化する処理。
- 既存の非標準プロジェクト（`CLAUDE.md` 欠落・単一 `.md`）への対応・移行。
- `wiki/` への ingest 連携（別系統のため対象外）。

## 4. 詳細設計

### 4.1 処理フロー

```
1. 対象ファイル特定
   - 起動文からファイル名/日付/タイトルを拾う → 05-meeting/ 内で解決。
   - 取れなければ 05-meeting/ の最新更新ファイルを候補提示して確認。
2. トランスクリプト検証
   - ファイルを読み、`# Teamsトランスクリプト` / `# Teams議事録` 見出し以下（または末尾）に
     本文があるか確認。空・欠落なら中止し、貼り付けを促す。
   - 構造化セクションや `# 議事録` が既に埋まっていれば上書き確認。
3. 議事録生成（in-file 編集）
   - 構造化セクションと `# 議事録` 本文を書き込む（§4.2）。トランスクリプトは残す。
4. プロジェクト推定（§4.3）
   - 01-projects/{work,personal}/ のフォルダ型プロジェクトを列挙し CLAUDE.md 概要/code と照合。
5. 確認（AskUserQuestion）
   - 候補（信頼度つき）を提示。複数選択可・「該当なし」可。
6. LOG/INDEX 更新（確認された各プロジェクト, §4.4）＋ mtg frontmatter に project リンク追加。
7. 報告（§4.6）
```

### 4.2 ミーティングノートへの書き込み

対象は `05-meeting/{YYYY-MM-DD}-{タイトル}.md`。`_templates/meeting.md` 由来の構造を持つ。

- **frontmatter**: 既存（`title`/`date`/`time`/`tags`）を保持し、`project` を追加（§4.5）。
- **`# 会議情報` の `参加者`**: トランスクリプトの話者名（`氏名 / 所属  時刻` 形式の行頭）から重複排除して列挙。
- **`## 議題`**: 主要トピックを箇条書き。
- **`## 決定事項`**: 決まったことを箇条書き（テンプレの空 `- ` 行を置換）。
- **`### アクションアイテム`**: `- [ ] {担当}: {内容}（{期限}）` 形式。担当/期限が不明なら省略。
- **`### 次回の予定`**: 次回日程・宿題があれば記載。無ければ空のまま。
- **`# 議事録`**: ナラティブ本文。既存ノート `2026-06-23-京都製作所.md` のスタイル（`■背景` / `■結論・今後のアクション` / 論点ごとの小見出し・箇条書き）に倣う。トランスクリプトに無い事実は書かない。
- **トランスクリプト本文は削除しない**（`# Teamsトランスクリプト` 以下をそのまま残す）。

テンプレートに `# 議事録` 見出しが無い版（`meeting.md` 現行はトランスクリプト見出しで終わる）の場合は、トランスクリプト見出しの**直前**に `# 議事録` セクションを挿入する。

### 4.3 プロジェクト推定

- `01-projects/work/` と `01-projects/personal/` 直下の**ディレクトリ**を候補とする（単一 `.md` ファイルは対象外）。
- 各候補の `CLAUDE.md` を読み、`概要` 本文と frontmatter `code`（例 `KT569`）を取得する。
- 議事録の `議題`/`決定事項`/`# 議事録` 本文中のキーワード・固有名詞・`code` と照合し、関連度の高い順に候補を作る（信頼度: 高/中/低）。
- 推定はするが**自動確定しない**。次工程で必ず確認する。

### 4.4 `LOG.md` / `INDEX.md` の更新

確認された各プロジェクト `01-projects/{type}/{name}/` に対して:

- `LOG.md`/`INDEX.md` が**無ければ** `new-project` 書式（当該 spec §4.2）で新規作成してから追記する。
- **`LOG.md`**:
  - `## ログ` の直下（先頭・最新が上）に1行追記:
    `- {会議日}: {会議の1行要約} → [[{mtgノート名}]]`
    （`{mtgノート名}` は拡張子なしのファイル名。`05-meeting/` のノート名は一意なので basename wikilink で解決する。）
  - 今回の決定・次アクションが現状を動かすなら `## 現状 / 次の一手` を更新（任意・内容次第）。
  - frontmatter `updated` を当日に更新。
- **`INDEX.md`**:
  - `## ミーティング` 見出しが無ければ末尾に作成し、`- [[{mtgノート名}]]` を追加（既出ならスキップ）。
  - frontmatter `updated` を当日に更新。

### 4.5 ミーティングノート frontmatter の `project` リンク

- frontmatter に `project` を追加する。値は各プロジェクトの `CLAUDE.md` を指す**パス修飾 wikilink**:
  - 例: `project: "[[01-projects/work/FHV7-AI向け良品学習アルゴ商品化支援/CLAUDE|FHV7-AI向け良品学習アルゴ商品化支援]]"`
  - パス修飾にする理由: 全プロジェクトが同名の `CLAUDE.md` を持つため、basename wikilink（`[[CLAUDE]]`）は vault 全体で曖昧（`new-project` spec §5 参照）。フルパス＋表示名で一意かつ可読にする。
- 複数プロジェクト該当時は YAML リストで複数持つ。
- 既に `project` がある場合は重複を避けて追加（マージ）。

### 4.6 報告

- 生成した議事録の要点（議題・主要決定・アクション数）。
- 更新したプロジェクトと編集ファイルのパス（`LOG.md`/`INDEX.md`、新規作成したかどうか）。
- mtg ノートに追加した `project` リンク。
- スキップした項目（該当プロジェクトなし、トランスクリプト未貼付け等）。

### 4.7 スキルファイル構成

```
.claude/skills/meeting-minutes/
└─ SKILL.md   ← frontmatter(name/description) + 手順
```

- 既存スキル（`new-project`）の流儀に合わせる。
- `description` は日本語の要約＋英語/日本語トリガーフレーズ（「議事録作って」「トランスクリプトから議事録」「mtg の議事録を生成して」「/meeting-minutes」等）を含める。

## 5. エッジケース・注意点

- **トランスクリプト無し/空**: 議事録を作れないので中止し、貼り付けを促す。
- **既存議事録あり**: 構造化セクションや `# 議事録` が埋まっている場合は上書き前に確認。
- **対象ファイル未指定**: `05-meeting/` の最新更新ファイルを候補提示して確認（明示渡しが基本）。
- **プロジェクト該当なし**: 議事録生成のみで完了。`LOG/INDEX` 更新と `project` リンク追加はスキップし、その旨を報告。
- **複数プロジェクト該当**: 各プロジェクトを更新。`project` リンクは複数持つ。
- **`LOG.md`/`INDEX.md` 不在**: `new-project` 書式で作成してから追記。
- **wikilink の曖昧性**: `05-meeting/` のノート名は一意 → LOG/INDEX → mtg は basename wikilink でよい。mtg → プロジェクト（`CLAUDE.md`）は同名衝突のためパス修飾 wikilink を用いる（§4.5）。
- **日付**: ハードコードせず、`date` は mtg ノート frontmatter の `date`（会議日）を、`updated` は処理日（当日）を用いる。
- **捏造防止**: トランスクリプトに無い決定・数値・固有名詞を補わない。

## 6. 受け入れ基準（Acceptance Criteria）

1. 対象ファイルを指定して起動すると、そのノートにトランスクリプトがある場合に議事録（構造化セクション＋`# 議事録` 本文）が生成され、**トランスクリプト本文は残る**。
2. トランスクリプトが空/欠落の場合、書き込まずに中止し、理由を報告する。
3. 議事録内容から `01-projects/{work,personal}/` のフォルダ型プロジェクトが推定され、AskUserQuestion で確認（複数選択・「該当なし」可）してから更新が走る。確認前に LOG/INDEX を書き換えない。
4. 確認された各プロジェクトの `LOG.md` の `## ログ` 先頭に `- {会議日}: {要約} → [[mtgノート名]]` が追記され、`INDEX.md` の `## ミーティング` に `- [[mtgノート名]]` が追加される。両ファイルの `updated` が当日に更新される。
5. 対象プロジェクトに `LOG.md`/`INDEX.md` が無い場合、`new-project` 書式で作成されたうえで上記追記が行われる。
6. ミーティングノート frontmatter に、各該当プロジェクトの `CLAUDE.md` を指すパス修飾 wikilink の `project` が追加される（複数該当時はリスト）。
7. 該当プロジェクトが無い場合、議事録のみ生成し、LOG/INDEX 更新と `project` 追加をスキップした旨を報告する。
8. 処理後、生成議事録の要点・更新ファイルパス・スキップ項目が報告される。

- 「未処理ミーティング（トランスクリプトはあるが議事録未生成）」の検出・一覧化。
